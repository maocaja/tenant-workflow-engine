Tenant Workflow Engine — Architecture

Architecture Reference · Learning Lab

Tenant Workflow Engine

A billing-document workflow shared across many clients, where the rules that differ per client are data, not code, every state change writes an immutable audit record, and approvals leave the process as events another service consumes.

Rules are rows, not ifs Every transition is audited Domain has zero Spring imports Events cross services exactly-once in effect

01 Ports & Adapters

A pure core, technology at the edges

The domain and the use cases know nothing about Spring, JPA, HTTP, or SQS — they depend only on ports (interfaces). Infrastructure adapters implement those ports. That inversion is what lets the core be unit-tested with hand-rolled fakes, no ApplicationContext to boot.

02 State machine

Four states, one rejection branch

Each arrow runs the same path: evaluate the tenant's rules for that gate → transition → write an audit event. Rules are scoped per gate, so an ApprovalGate (signatures required) is checked at approve, not at submit — when the signatures don't exist yet.

03 The distributed path

From an approval to a second service — without losing or double-counting

Approving does three writes in one database transaction: the document, the audit event, and an outbox row. Because the event is a row in the same database, there is no dual-write gap. A relay ships outbox rows to SQS after commit (at-least-once); the consumer dedups on the event id so the effect is exactly-once. A message that keeps failing lands in the dead-letter queue.

05 Target topology

Where it runs on AWS

Locally the same code runs against Colima (Postgres) and LocalStack (SQS). In the cloud, only the endpoints change: containers on ECS Fargate, managed RDS Postgres, real SQS with a redrive policy onto a DLQ. Nothing in the application code is AWS-specific beyond the SQS client's endpoint.

06 Run it

Two services, one broker, two databases

Infrastructure

colima start · Postgres on :55432 (dbs workflow + consumer) · LocalStack SQS on :4566.

Producer

gradlew bootRun --args=--spring.profiles.active=local → REST on :8080.

Consumer

gradlew :consumer-service:bootRun → polls SQS on :8081.

# the flow, over HTTP
POST /api/documents            → DRAFT
POST /api/documents/{id}/submit    → rules(SUBMIT) → SUBMITTED | REJECTED
POST /api/documents/{id}/signatures  → +1 signature
POST /api/documents/{id}/approve   → rules(APPROVE) → APPROVED (+ outbox event) | REJECTED
GET  /api/documents/{id}       → document + full audit history
