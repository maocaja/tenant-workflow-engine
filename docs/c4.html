Tenant Workflow Engine — C4

The same system at three zoom levels. Each level answers a different question, for a different reader: the whole system in its world, the runnable pieces inside it, and the parts inside one piece.

1 · Context 2 · Container 3 · Component 4 · Code → the repo itself
Person / actor Container (deployable/runnable) Database Message queue External system

Level 1 System Context

The system in its world

The highest zoom: one box for the whole platform, who uses it, and what it talks to. A non-technical stakeholder should understand this. Here: an API caller drives documents through the workflow, and the platform emits approval events that downstream systems (payments, notifications) consume.

Level 2 Containers

The runnable pieces inside the platform

Zoom into the box. A container is something that runs and holds state — a service, a database, a queue — each with its technology. This is the most useful view for an engineer: it shows the two services, their separate databases, and the queue with its dead-letter queue. Notice each service owns its own database — no shared schema.

Level 3 Components

Inside workflow-service

Zoom into one container. Components are the major groupings of code and how they depend on each other — this is the hexagonal / ports-and-adapters structure. Dependencies point inward: infrastructure → application → domain, and the domain depends on nothing.

Level 4 Code

The repository is the fourth level

C4 stops at code because the code is the code — you read it, you don't diagram it. Component Domain is src/main/java/com/mauricio/workflow/domain/; Use cases is application/; the adapters live under infrastructure/. The consumer is its own module, consumer-service/.
