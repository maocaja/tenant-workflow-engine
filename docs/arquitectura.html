Motor de Workflow Multi-tenant

Un flujo de facturas compartido por muchos clientes, donde las condiciones que cambian por cliente son datos, no código, cada cambio de estado escribe un registro de auditoría inmutable, y las aprobaciones salen del proceso como eventos que otro servicio consume.

Las reglas son filas, no ifs Cada transición se audita El dominio no importa Spring Los eventos cruzan servicios sin duplicar efecto

01 Puertos y Adaptadores

Un núcleo puro, la tecnología en los bordes

El dominio y los casos de uso no saben nada de Spring, JPA, HTTP ni SQS — solo dependen de puertos (interfaces). Los adaptadores de infraestructura implementan esos puertos. Esa inversión es lo que permite testear el núcleo con fakes a mano, sin arrancar un ApplicationContext.

02 Máquina de estados

Cuatro estados, una rama de rechazo

Cada flecha corre el mismo camino: evalúa las reglas del cliente para ese gate → transiciona → escribe un evento de auditoría. Las reglas están scopeadas por gate, así que una ApprovalGate (firmas requeridas) se verifica al aprobar, no al enviar — cuando las firmas todavía no existen.

03 El camino distribuido

De una aprobación a un segundo servicio — sin perder ni duplicar

Aprobar hace tres escrituras en una sola transacción de base de datos: el documento, el evento de auditoría y una fila de outbox. Como el evento es una fila en la misma base, no hay hueco de dual-write. Un relay envía las filas del outbox a SQS después del commit (al menos una vez); el consumidor deduplica por id de evento, así el efecto es exactamente una vez. Un mensaje que sigue fallando termina en la cola de mensajes muertos (DLQ).

05 Topología objetivo

Dónde corre en AWS

Localmente el mismo código corre contra Colima (Postgres) y LocalStack (SQS). En la nube solo cambian los endpoints: contenedores en ECS Fargate, RDS Postgres gestionado, SQS real con redrive a una DLQ. Nada en el código de la aplicación es específico de AWS más allá del endpoint del cliente SQS.

06 Cómo correrlo

Dos servicios, un broker, dos bases de datos

Infraestructura

colima start · Postgres en :55432 (dbs workflow + consumer) · LocalStack SQS en :4566.

Productor

gradlew bootRun --args=--spring.profiles.active=local → REST en :8080.

Consumidor

gradlew :consumer-service:bootRun → hace poll a SQS en :8081.

# el flujo, por HTTP
POST /api/documents            → DRAFT
POST /api/documents/{id}/submit    → reglas(SUBMIT) → SUBMITTED | REJECTED
POST /api/documents/{id}/signatures  → +1 firma
POST /api/documents/{id}/approve   → reglas(APPROVE) → APPROVED (+ evento outbox) | REJECTED
GET  /api/documents/{id}       → documento + historial de auditoría completo
